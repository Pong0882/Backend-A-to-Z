## 왜 Redis로 전환하는가

현재 Refresh Token은 MySQL `refresh_tokens` 테이블에 저장되어 있다.

**DB 방식의 문제:**
- 토큰 조회/저장마다 디스크 I/O 발생
- 만료된 토큰이 자동 삭제되지 않고 계속 쌓임
- 만료 체크를 코드에서 직접 해야 함 (`expiresAt.isBefore(now())`)

**Redis로 바꾸면:**
- 메모리 기반 → 훨씬 빠름
- TTL 설정 시 만료된 토큰 자동 삭제
- 코드가 단순해짐

**목적:** DB vs Redis 성능 비교 (k6로 로그인 반복 후 P95 응답시간 측정)

---

## 설계 — 인터페이스로 추상화

그냥 Redis로 교체하지 않고, 두 구현체를 모두 남겨서 설정값 하나로 전환할 수 있게 설계했다.

```
RefreshTokenStore (인터페이스)
├── DbRefreshTokenStore   → MySQL (기존 방식)
└── RedisRefreshTokenStore → Redis (신규)

application.yml의 token.store 값으로 전환:
  token.store: db    → DbRefreshTokenStore 사용
  token.store: redis → RedisRefreshTokenStore 사용
```

**이렇게 설계한 이유:**
- 성능 비교를 위해 두 구현체를 동시에 유지해야 함
- 나중에 Redis로 완전 전환 결정 시 설정값만 바꾸면 됨
- `AuthService`가 구현체를 몰라도 됨 (인터페이스에만 의존)

## 구현

### RefreshTokenStore 인터페이스

```java
public interface RefreshTokenStore {
    void save(String email, String token, long expirationMs);
    Optional<String> findEmailByToken(String token);
    void deleteByEmail(String email);
}
```

### DbRefreshTokenStore

기존 JPA 방식 그대로. `expiresAt` 컬럼으로 만료 여부 직접 체크.

```java
@Override
public Optional<String> findEmailByToken(String token) {
    return refreshTokenRepository.findByToken(token)
            .filter(rt -> rt.getExpiresAt().isAfter(LocalDateTime.now()))
            .map(RefreshToken::getEmail);
}
```

### RedisRefreshTokenStore

두 가지 키로 양방향 매핑:

```
refresh:token:{token}  → 값: email   (토큰으로 이메일 조회)
refresh:email:{email}  → 값: token   (이메일로 토큰 삭제)
```

**왜 두 가지 키가 필요한가:**
- 재발급 시: 토큰 → 이메일 조회 필요 (`refresh:token:xxx`)
- 로그아웃 시: 이메일 → 토큰 삭제 필요 (`refresh:email:xxx`)
- Redis는 역방향 조회가 없어서 양방향 매핑으로 해결

```java
@Override
public void save(String email, String token, long expirationMs) {
    long expirationSec = expirationMs / 1000;

    // 기존 토큰 정리 (재로그인 시 이전 토큰 키 삭제)
    String oldToken = redisTemplate.opsForValue().get(EMAIL_KEY_PREFIX + email);
    if (oldToken != null) {
        redisTemplate.delete(TOKEN_KEY_PREFIX + oldToken);
    }

    redisTemplate.opsForValue().set(TOKEN_KEY_PREFIX + token, email, expirationSec, TimeUnit.SECONDS);
    redisTemplate.opsForValue().set(EMAIL_KEY_PREFIX + email, token, expirationSec, TimeUnit.SECONDS);
}
```

TTL은 `save()` 호출 시 `expirationMs`로 전달 → yml의 `jwt.refresh-token-expiration` 값 사용.

### RefreshTokenStoreConfig

```java
@Bean
public RefreshTokenStore refreshTokenStore(
        @Qualifier("dbRefreshTokenStore") RefreshTokenStore dbStore,
        @Qualifier("redisRefreshTokenStore") RefreshTokenStore redisStore
) {
    return switch (tokenStore) {
        case "redis" -> redisStore;
        default -> dbStore;
    };
}
```

### AuthService 변경

`RefreshTokenRepository` 직접 의존 → `RefreshTokenStore` 인터페이스 의존으로 교체.

**이전:**
```java
// 만료 체크 + 삭제 + 이메일 꺼내기를 AuthService에서 직접 처리
RefreshToken saved = refreshTokenRepository.findByToken(token).orElseThrow(...);
if (saved.getExpiresAt().isBefore(LocalDateTime.now())) {
    refreshTokenRepository.delete(saved);
    throw new ExpiredTokenException();
}
String email = saved.getEmail();
```

**이후:**
```java
// 한 줄로 끝남 — 만료 체크/삭제는 구현체가 처리
String email = refreshTokenStore.findEmailByToken(token).orElseThrow(...);
```

`ExpiredTokenException`도 제거 — Redis는 만료되면 키가 없는 것과 같으므로 `InvalidTokenException`으로 통일.

## 트러블슈팅

### Redis keys * 했는데 empty array

**상황:** `token.store: redis`로 설정했는데 Redis에 키가 없음

**원인:** 로컬에 Redis가 별도로 설치되어 있었음 (Windows 서비스로 실행 중, PID 7236)
Spring Boot가 `localhost:6379`로 연결하는데 Docker Redis가 아닌 로컬 Redis로 연결됐던 것

```bash
netstat -ano | findstr 6379
# TCP 0.0.0.0:6379  LISTENING  7236
tasklist | findstr 7236
# redis-server.exe  7236
```

**해결:** 로컬 Redis 서비스 삭제

```powershell
Stop-Service Redis
sc.exe delete Redis
```
