### users 테이블 설계 확정

#### nickname 처리 방식

회원가입 시 nickname을 받지 않고 서버에서 자동 생성하기로 결정.
형용사 + 명사 조합 랜덤 생성 방식 (예: `빠른토끼`, `조용한독수리`).
나중에 마이페이지에서 수정 가능.

**왜 랜덤 생성인가:**
- 회원가입 폼이 간단해짐 (email + password만 받으면 됨)
- 사용자가 닉네임 고민하는 마찰 제거
- 나중에 바꾸면 되니까 처음엔 그냥 배정

nickname은 `unique` 제약 필요 — 랜덤 생성 시 중복 체크 후 재생성 로직 필요.

#### 소셜 로그인 방식 결정 — A) users 통합 방식

**고민한 것:**

소셜 로그인 유저는 password가 없다. 이걸 어떻게 처리하느냐를 두고 두 가지 선택지를 비교했다.

**A) users 통합 방식**
```
users: email, password(nullable), nickname, ...
oauth_accounts: user_id, provider, provider_id, ...
```
- 같은 테이블에 일반 유저 + 소셜 유저가 공존
- 소셜 유저는 password = null

**B) 완전 분리 방식**
- 일반 유저 테이블 / 소셜 유저 테이블 따로
- Spring Security 인증 처리를 두 테이블 기준으로 따로 구현해야 함

**결정: A 방식**

**정규화 관점에서 A 방식이 완벽하지 않은 이유:**
- 소셜 유저의 `password` 컬럼이 null이 된다
- null 허용 컬럼은 엄밀한 의미에서 "모든 행이 해당 속성을 가져야 한다"는 관계형 모델 원칙에 어긋남
- 정규화 위반은 아니지만(함수 종속성 문제가 아니라 nullable 문제) 테이블이 두 가지 유형의 엔티티를 동시에 표현한다는 점에서 설계상 트레이드오프가 있음

**그럼에도 A 방식을 선택한 이유:**
1. **Spring Security 통합이 자연스럽다** — `UserDetailsService`가 `users` 테이블 하나만 바라보면 됨. 분리하면 일반/소셜 유저를 각각 다른 방식으로 인증해야 해서 필터/핸들러가 두 배로 복잡해짐
2. **같은 이메일로 일반+소셜 계정 연동 가능** — "카카오로 가입했는데 나중에 일반 로그인도 쓰고 싶다"는 요구사항을 자연스럽게 처리 가능. 분리 방식에선 이 연동 자체가 별도 기능이 됨
3. **실무 표준** — Spring Security OAuth2 공식 예제, 대부분의 실무 코드가 이 방식. null 허용 트레이드오프를 감수하는 게 업계 컨센서스
4. **null 의미가 명확하다** — `password = null`이면 "소셜 전용 유저"라는 의미가 코드 레벨에서 명확함. 오히려 테이블 분리하면 어느 테이블에서 왔는지 추적이 더 복잡해짐

**결론:** 정규화를 약간 희생하되, 구현 복잡도와 Spring Security 통합 편의성을 얻는 트레이드오프를 의도적으로 선택했다.

#### Soft Delete 선택 이유

**선택지:**
- Hard Delete: `DELETE FROM users WHERE id = ?` — 데이터 완전 삭제
- Soft Delete: `deleted_at` 컬럼에 삭제 시각 기록, 실제 행은 남김

**결정: Soft Delete**

**이유:**
1. **데이터 복구 가능** — 실수로 탈퇴하거나 어드민이 잘못 삭제했을 때 복구 가능. Hard Delete는 한 번 지우면 끝
2. **연관 데이터 무결성** — 탈퇴한 유저가 만든 주문/전략/게시글이 남아있는데 users 행이 사라지면 FK가 끊김. Soft Delete면 연관 데이터가 여전히 유저를 참조할 수 있음
3. **탈퇴 유저 분석 가능** — "언제 가입해서 언제 탈퇴했는지" 같은 유저 이탈 분석이 가능. 개인정보 보호법상 일정 기간 데이터 보관 의무가 있는 경우도 있음
4. **이메일 재가입 처리 용이** — 탈퇴 후 같은 이메일로 재가입할 때 Hard Delete면 그냥 INSERT, Soft Delete면 `deleted_at IS NOT NULL` 체크 후 복구하거나 새 행 생성 중 선택 가능

**트레이드오프:**
- 쿼리마다 `WHERE deleted_at IS NULL` 조건을 붙여야 함 → 실수하면 탈퇴 유저 데이터가 노출됨
- JPA에서 `@Where(clause = "deleted_at IS NULL")` 또는 `@SQLRestriction`으로 전역 필터 걸어서 해결 가능

#### oauth_accounts 설계 — ERD는 구현 순서와 무관하게 연관 테이블을 같이 잡아야 한다

처음에 oauth_accounts를 "나중에 소셜 로그인 구현할 때 추가하자"로 넘어가려 했다.
사용자가 "외부 고유 ID 같은 거 필요하지 않나?"라고 짚었고, 이게 맞는 판단이었다.

**왜 같이 설계해야 하나:**
- ERD 설계의 핵심은 지금 당장 구현 안 해도 관계는 미리 정의해두는 것
- users에 데이터가 쌓인 후 oauth_accounts를 추가하면 FK 관계 추가가 번거로워짐
- `login_type` 컬럼이 oauth_accounts 없이는 반쪽짜리 설계가 됨
- 구현 순서와 설계 순서를 혼동하면 안 된다 — 설계는 전체를 보고, 구현은 단계별로

**provider_id가 필요한 이유:**
카카오/구글로 로그인하면 공급자가 고유 ID를 넘겨줌 (`provider_id`).
이 ID로 "이미 가입한 유저인지" 판별함.
`users`에 `provider_id` 컬럼을 직접 넣으면 다중 소셜 연동(카카오+구글 동시)이 불가능.
`oauth_accounts`를 1:N으로 분리해야 한 유저가 여러 소셜 계정을 연동할 수 있음.

**oauth_accounts 확정:**
```sql
oauth_accounts
- id          BIGINT PK AUTO_INCREMENT
- user_id     BIGINT FK → users.id NOT NULL
- provider    ENUM('KAKAO', 'GOOGLE', 'NAVER') NOT NULL
- provider_id VARCHAR(100) NOT NULL
- created_at  DATETIME NOT NULL

UNIQUE (provider, provider_id)  -- 같은 카카오 계정이 두 번 등록되는 것 방지
```

**users 최종 확정:**
```sql
users
- id            BIGINT PK AUTO_INCREMENT
- email         VARCHAR(100) UNIQUE NOT NULL
- password      VARCHAR(255) NULL            -- 소셜 유저는 null
- nickname      VARCHAR(50) UNIQUE NOT NULL  -- 회원가입 시 랜덤 자동 생성
- profile_image VARCHAR(500) NULL
- point_balance BIGINT DEFAULT 0 NOT NULL
- role          ENUM('ROLE_USER', 'ROLE_ADMIN') NOT NULL DEFAULT 'ROLE_USER'
- login_type    ENUM('LOCAL', 'SOCIAL') NOT NULL DEFAULT 'LOCAL'
- is_active     BOOLEAN DEFAULT TRUE NOT NULL
- deleted_at    DATETIME NULL                -- Soft Delete
- created_at    DATETIME NOT NULL
- updated_at    DATETIME NOT NULL
```

#### broker_accounts 분리 결정

증권사 계좌(KIS API 키)를 `users` 테이블에 컬럼으로 넣지 않고 `broker_accounts`로 분리.

**이유:**
- 계좌가 여러 개일 수 있음 (모의투자 계좌 + 실전 계좌 → 1:N 관계)
- KIS API 키(appkey, appsecret)가 같이 붙는데 users에 넣으면 테이블이 비대해짐
- 나중에 증권사가 추가되면 `broker_type` 컬럼으로 확장 가능
- 정규화 관점에서도 API 키는 users의 속성이 아니라 별도 엔티티

**KIS access_token 저장 위치 결정 — Redis**

현재 `KisAuthService`는 access_token을 JVM 힙 메모리(싱글톤 Bean 필드)에 캐싱하고 있음.
코드 주석에도 "추후 다중 사용자 지원 시 Redis로 확장 예정"이라고 적혀 있었음.

저장 위치 후보를 비교했다:

| 위치 | 판단 |
|------|------|
| JVM 메모리 (현재) | 서버 재시작 시 날아감. 단일 계정이면 동작하지만 다중 유저 구조 불가 |
| broker_accounts (MySQL) | 영구 저장 목적. 만료가 있는 임시 토큰을 DB에 넣는 건 맞지 않음 |
| MongoDB | 구조가 자주 바뀌는 데이터에 적합. KIS 토큰은 구조가 단순하고 고정적 → 굳이 MongoDB 쓸 이유 없음. 인프라 추가 비용도 발생 |
| Redis | 임시 데이터, TTL 자동 만료, 다중 유저 `kis:token:{userId}` 키 매핑 자연스러움. 이미 Refresh Token용으로 쓰고 있어서 추가 인프라 없음 |

**결정:**

| 데이터 | 저장 위치 | 이유 |
|--------|-----------|------|
| appkey / appsecret | `broker_accounts` (MySQL) | 영구 저장, 암호화 필요 |
| access_token | Redis (`kis:token:{userId}`) | 임시 데이터, TTL 자동 만료, 다중 유저 지원 |

`broker_accounts`에 access_token 컬럼을 두지 않기로 확정.

**broker_accounts 설계 고민 — broker_type 단일 컬럼 vs 분리**

처음에 `broker_type ENUM('KIS_MOCK', 'KIS_REAL')`로 설계했는데 증권사 확장을 고려하면서 재검토했다.

**문제:** 증권사가 추가될 때마다 ENUM에 값을 추가해야 해서 `ALTER TABLE` 필요.

**선택지:**

A) broker + account_type 컬럼 분리
```sql
broker       ENUM('KIS', 'KIWOOM', 'SAMSUNG')
account_type ENUM('MOCK', 'REAL')
UNIQUE (user_id, broker, account_type)
```

B) brokers 별도 테이블 분리
```sql
brokers 테이블: id, code, name
broker_accounts: broker_id FK → brokers.id
```
증권사 추가 시 ALTER 없이 INSERT만 하면 됨. 대신 JOIN 추가.

**결정: A 방식**

이유:
- 증권사 목록은 적고 자주 바뀌지 않아서 ENUM으로 관리해도 충분
- `brokers` 테이블까지 만들면 JOIN이 하나 더 생기는데 그만한 이득이 없음
- 지금 단계에서 B는 오버엔지니어링
- ENUM 두 개 조합으로 의미가 명확하고 `UNIQUE (user_id, broker, account_type)`으로 중복 방지도 깔끔

**KIS는 계정당 appkey를 하나씩만 발급** → `UNIQUE (user_id, broker, account_type)` 제약이 실제 비즈니스 규칙과도 일치.

**broker_accounts 최종 확정:**
```sql
broker_accounts
- id           BIGINT PK AUTO_INCREMENT
- user_id      BIGINT FK → users.id NOT NULL
- broker       ENUM('KIS', 'KIWOOM', 'SAMSUNG') NOT NULL
- account_type ENUM('MOCK', 'REAL') NOT NULL
- appkey       VARCHAR(100) NOT NULL   -- 암호화 저장
- appsecret    VARCHAR(200) NOT NULL   -- 암호화 저장
- is_active    BOOLEAN DEFAULT TRUE NOT NULL
- created_at   DATETIME NOT NULL
- updated_at   DATETIME NOT NULL

UNIQUE (user_id, broker, account_type)
```

### stocks / stock_prices 테이블 보강

#### stocks — market 컬럼 추가 및 정규화 고민

현재 `stocks`에 `code UNIQUE`, `name`만 있음. 미국 주식까지 다루려면 시장 구분이 필요해서 `market` 컬럼 추가를 논의했다.

**문제 제기:**
`market` 추가 시 `UNIQUE (code, market)`으로 바뀌는데, `name`이 `(code, market)` 복합키에 종속되는 구조가 됨.
실제로 `name`은 `code`만으로 결정되므로 (삼성전자는 KRX든 어디든 삼성전자) 이는 2NF 위반.

**해결 방법으로 검토한 것:**
```sql
-- 종목 기본 정보 (시장 무관)
stocks: id, code UNIQUE, name

-- 시장별 상장 정보
stock_listings: id, stock_id FK, market
UNIQUE (stock_id, market)
```

**결정: A) stocks에 market 컬럼 추가 + UNIQUE(code, market)**

이유:
- 현실적으로 같은 종목코드가 여러 시장에 상장되는 케이스가 거의 없음
- 국내(KRX)와 미국(NASDAQ/NYSE)은 코드 체계 자체가 달라서 실제로 겹치지 않음 (국내는 숫자 6자리, 미국은 알파벳)
- `stock_listings` 분리는 이론상 완벽하지만 JOIN이 추가되고 지금 단계에서 오버엔지니어링
- 2NF 위반을 인지하고 의도적으로 트레이드오프를 선택한 것

**stocks 최종 확정:**
```sql
stocks
- id      BIGINT PK AUTO_INCREMENT
- code    VARCHAR(20) NOT NULL
- name    VARCHAR(50) NOT NULL
- market  ENUM('KRX', 'NASDAQ', 'NYSE') NOT NULL

UNIQUE (code, market)   -- 기존 code UNIQUE → (code, market) UNIQUE로 변경
```

#### stock_prices — 가격 타입 변경

기존 `BIGINT`로 선언된 가격 컬럼들을 `DECIMAL(12, 4)`로 변경.

이유:
- 미국 주식은 달러 단위 소수점 있음 (예: `182.3400`)
- 국내 주식은 원화라 소수점 없지만 DECIMAL로 저장해도 문제없음 (`.0000`으로 저장)
- 나중에 타입 변경하면 데이터 마이그레이션 비용 발생 → 지금 잡는 게 맞음

**stock_prices 최종 확정:**
```sql
stock_prices
- id         BIGINT PK AUTO_INCREMENT
- stock_id   BIGINT FK → stocks.id NOT NULL
- trade_date DATE NOT NULL
- open_price  DECIMAL(12, 4) NOT NULL
- high_price  DECIMAL(12, 4) NOT NULL
- low_price   DECIMAL(12, 4) NOT NULL
- close_price DECIMAL(12, 4) NOT NULL
- volume      BIGINT NOT NULL

UNIQUE (stock_id, trade_date)
```

### watchlists 테이블 설계

관심 종목 알림을 어디서 관리할지 먼저 정리했다.

**자동매매 체결 알림 vs 관심 종목 가격 알림 구분:**
- 자동매매 체결/실패 알림 → `notifications` 테이블 (나중에 설계)
- 관심 종목 가격 알림 ("삼성전자 8만원 되면 알려줘") → `watchlists.alert_price`

`watchlists`에 `alert_price` 컬럼 하나로 충분한 이유:
- 지금 단계에서 가격 알림 조건이 단순함 (목표가 하나)
- 조건이 복잡해지면 나중에 `stock_alerts` 테이블로 분리 가능
- 지금 `stock_alerts`까지 만드는 건 오버엔지니어링

**watchlists 확정:**
```sql
watchlists
- id          BIGINT PK AUTO_INCREMENT
- user_id     BIGINT FK → users.id NOT NULL
- stock_id    BIGINT FK → stocks.id NOT NULL
- alert_price DECIMAL(12, 4) NULL  -- 목표가 알림 (null이면 알림 없음)
- created_at  DATETIME NOT NULL

UNIQUE (user_id, stock_id)  -- 같은 종목 중복 등록 방지
```

### 전략 실행 방식 결정 — 스케줄러 vs 실시간 WebSocket

**선택지:**
- 스케줄러 (1분마다 조건 체크)
- 실시간 WebSocket (시세 수신 즉시 조건 체크)

**결정: 스케줄러로 시작, 나중에 WebSocket으로 개선**

판단 근거:
- 스케줄러는 `@Scheduled` 하나로 구현 가능. 지금 단계에서 WebSocket 연결 관리까지 복잡도를 올릴 필요 없음
- PROJECT.md 철학 — "처음부터 최적화하지 않는다. 느려지는 지점을 직접 체감하고 단계적으로 개선한다"
- 스케줄러로 돌리다가 "1분 지연이 문제다"를 직접 체감한 다음 WebSocket으로 바꾸는 게 학습 효과가 훨씬 큼
- 스케줄러용 컬럼 (`status`, `last_checked_at`)은 WebSocket 전환 후에도 그대로 유효 → 호환성 문제 없음

### strategies + strategy_conditions 테이블 설계

#### 전략 실행 단위 — 수량 방식 선택

1회 주문 단위를 금액/수량/비율 중 선택해야 했다.

- 금액 방식: 주가에 따라 실제 매수 수량이 달라져서 매번 계산 필요
- 비율 방식: 포트폴리오 잔액을 실시간 조회해야 해서 복잡
- **수량 방식**: "10주 매수" → KIS API에 수량 그대로 전달. 단순하고 명확

**결정: 수량 방식 (`order_quantity INT`)**

**strategies 확정:**
```sql
strategies
- id                 BIGINT PK AUTO_INCREMENT
- user_id            BIGINT FK → users.id NOT NULL
- broker_account_id  BIGINT FK → broker_accounts.id NOT NULL
- stock_id           BIGINT FK → stocks.id NOT NULL
- name               VARCHAR(100) NOT NULL
- status             ENUM('ACTIVE', 'INACTIVE', 'PAUSED') NOT NULL DEFAULT 'INACTIVE'
- order_quantity     INT NOT NULL
- last_checked_at    DATETIME NULL
- created_at         DATETIME NOT NULL
- updated_at         DATETIME NOT NULL
```

#### strategy_conditions — MySQL JSON 타입 사용 가능 여부

MySQL 5.7.8부터 JSON 타입 공식 지원. `JSON_EXTRACT()`로 조회도 가능하지만 JSON 컬럼에는 인덱스를 직접 걸 수 없음.

`strategy_conditions`에서 `params` 내부 값으로 쿼리할 일이 거의 없음 (보통 `strategy_id`로 조건 전체를 가져와서 Java에서 처리). `strategy_id`에만 인덱스 걸면 충분.

#### 지원할 조건 유형 목록

1단계 구현 대상:
```
PRICE_BELOW / PRICE_ABOVE       가격 조건
PROFIT_TARGET                   목표 수익률
STOP_LOSS                       손절
RSI_BELOW / RSI_ABOVE           RSI 지표
MA_CROSS_GOLDEN / MA_CROSS_DEAD 골든/데드크로스
```

나중에 추가 가능한 것들:
```
PRICE_DROP      전일 대비 N% 하락
TRAILING_STOP   고점 대비 N% 하락 시 매도
VOLUME_SURGE    거래량 급등
BOLLINGER_LOWER/UPPER  볼린저밴드
MACD_CROSS      MACD 시그널 크로스
```

`indicator VARCHAR(50)`이라 나중에 새 조건 추가 시 테이블 변경 없이 값만 추가하면 됨.

**strategy_conditions 확정:**
```sql
strategy_conditions
- id          BIGINT PK AUTO_INCREMENT
- strategy_id BIGINT FK → strategies.id NOT NULL
- type        ENUM('BUY', 'SELL') NOT NULL
- indicator   VARCHAR(50) NOT NULL
- params      JSON NOT NULL
- created_at  DATETIME NOT NULL
```

params 예시:
```json
RSI_BELOW    → {"period": 14, "threshold": 30}
MA_CROSS     → {"short": 5, "long": 20}
PROFIT_TARGET → {"percent": 5.0}
STOP_LOSS    → {"percent": -3.0}
```

### orders + order_executions 테이블 설계

#### 주문과 체결을 분리하는 이유

주문 1개가 여러 번에 나눠서 체결될 수 있음.
예: 삼성전자 100주 주문 → 50주 체결 → 30주 체결 → 20주 체결
→ `orders` 1 : N `order_executions` 구조로 분리.

#### 시장가 vs 지정가 — 둘 다 지원

처음엔 시장가만 해도 충분하지 않냐는 의견이 있었지만 지정가도 함께 지원하기로 결정.

**이유:**
- 시장가만 하면 슬리피지 발생 (원하는 가격과 실제 체결 가격 차이)
- 실제 자동매매에서 지정가가 훨씬 중요함
- 지정가는 미체결 관리가 필요하지만 (얼마나 기다릴지, 취소 시점) 이건 서비스 로직에서 처리
- `price_type` 컬럼 하나로 두 방식을 깔끔하게 구분 가능

**orders 확정:**
```sql
orders
- id                BIGINT PK AUTO_INCREMENT
- user_id           BIGINT FK → users.id NOT NULL
- strategy_id       BIGINT FK → strategies.id NULL     -- 자동매매면 있고 수동이면 null
- broker_account_id BIGINT FK → broker_accounts.id NOT NULL
- stock_id          BIGINT FK → stocks.id NOT NULL
- order_type        ENUM('BUY', 'SELL') NOT NULL
- price_type        ENUM('MARKET', 'LIMIT') NOT NULL    -- 시장가/지정가
- status            ENUM('PENDING', 'PARTIAL', 'FILLED', 'CANCELLED', 'FAILED') NOT NULL
- quantity          INT NOT NULL                         -- 주문 수량
- price             DECIMAL(12, 4) NULL                  -- 지정가일 때만 값 있음
- filled_quantity   INT DEFAULT 0 NOT NULL               -- 누적 체결 수량
- created_at        DATETIME NOT NULL
- updated_at        DATETIME NOT NULL
```

**order_executions 확정:**
```sql
order_executions
- id          BIGINT PK AUTO_INCREMENT
- order_id    BIGINT FK → orders.id NOT NULL
- quantity    INT NOT NULL                 -- 이번에 체결된 수량
- price       DECIMAL(12, 4) NOT NULL      -- 실제 체결 가격
- executed_at DATETIME NOT NULL
```

### portfolios + holdings 테이블 설계

#### portfolios 테이블을 별도로 두는 이유

유저당 포트폴리오가 하나면 `portfolios` 테이블 없이 `holdings`에 `user_id` 직접 걸어도 동작함.
그럼에도 `portfolios` 테이블을 별도로 두기로 결정.

**이유:**
1. 나중에 포트폴리오 여러 개로 확장할 때 `holdings` 테이블 건드릴 필요 없음. 지금 `user_id` 직접 걸면 확장 시 컬럼 추가 + 데이터 마이그레이션 필요
2. 포트폴리오 자체에 메타데이터(`name`, `description`) 붙이기 쉬움
3. `users → portfolios → holdings` 흐름이 도메인 모델로서 자연스러움
4. 테이블 하나 더 있는 비용보다 나중에 없어서 후회하는 비용이 더 큼

#### 종목 숨기기 기능

포트폴리오 화면에서 특정 종목을 숨길 수 있어야 한다는 요구사항 추가.
`holdings`에 `is_hidden` 컬럼으로 처리. 조회 시 기본 `WHERE is_hidden = FALSE`, 숨긴 종목 포함 시 조건 제거.

**portfolios 확정:**
```sql
portfolios
- id         BIGINT PK AUTO_INCREMENT
- user_id    BIGINT FK → users.id NOT NULL UNIQUE
- created_at DATETIME NOT NULL
- updated_at DATETIME NOT NULL
```

**holdings 확정:**
```sql
holdings
- id             BIGINT PK AUTO_INCREMENT
- portfolio_id   BIGINT FK → portfolios.id NOT NULL
- stock_id       BIGINT FK → stocks.id NOT NULL
- quantity       INT NOT NULL
- average_price  DECIMAL(12, 4) NOT NULL
- is_hidden      BOOLEAN DEFAULT FALSE NOT NULL
- created_at     DATETIME NOT NULL
- updated_at     DATETIME NOT NULL

UNIQUE (portfolio_id, stock_id)
```

### 미래 확장 — AI/ML 연동 계획

ERD 설계 중 나중에 붙일 AI 기능들을 논의했다. 지금 MySQL ERD와 충돌 없이 단계별로 추가 가능.

#### 저장소 선택 기준

AI/ML 관련 데이터는 성격에 따라 저장소가 달라진다:

| 데이터 | 저장소 | 이유 |
|--------|--------|------|
| 뉴스 스크랩 + 감정분석 | Elasticsearch | 텍스트 검색 최적화, 감정 점수 집계 |
| AI 대화 히스토리 | MongoDB | 비정형, 스키마 유연, 대화 맥락 저장 |
| AI 예측 결과 | MySQL | 정형 데이터, 이력/정확도 관리 |
| 실시간 시세 캐싱 | Redis | TTL, 빠른 읽기 |

#### 단계별 AI 확장 로드맵

```
STEP 6 — AI 연동
  뉴스 스크랩 + ES 저장
  → 뉴스 감정 분석 (Python 서비스)
  → AI Agent (Claude MCP) — 종목 분석, 전략 최적화, 자연어 → 전략 변환
  → 백테스팅 엔진 (stock_prices 과거 데이터 기반)

STEP 7 — MLOps 자동화
  ML 모델 학습/추론 서비스 (Python — PyTorch/scikit-learn)
  → 모델 자동 재학습 파이프라인 (Kafka 이벤트 트리거)
     매일 새 시세 데이터 → 학습 데이터 업데이트 → 재학습 → 백테스팅 → 자동 교체
  → MLflow 모델 버전 관리
  → Airflow 파이프라인 스케줄링
  → Harness 테스트 (PR 머지 시 백테스팅 자동 실행, 수익률 기준 미달 시 머지 차단)
```

### 전략 조건 저장 방식 결정 — 컬럼 분리 vs JSON vs 혼합

**선택지:**
- A) 컬럼 분리: `indicator`, `operator`, `value`, `period` 컬럼으로 분리
- B) JSON 컬럼 하나에 전부: `conditions JSON`
- C) 혼합: `indicator` 컬럼 + `params JSON`

**결정: C) 혼합 방식**

판단 근거:
- 지금 전략 조건 종류가 확정되지 않았음. 컬럼으로 다 잡으면 새 조건 추가 시마다 `ALTER TABLE` 반복
- 그렇다고 JSON 하나에 다 박으면 "RSI 조건 사용 중인 전략이 몇 개야?" 같은 쿼리 불가능
- `indicator` 컬럼으로 조건 유형을 잡고 세부 파라미터(`period`, `threshold` 등)는 `params JSON`으로 유연하게
