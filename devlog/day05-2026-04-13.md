### [Bug #6] 로그아웃 API 인증 없이 200 반환 수정

**문제:**
JWT 토큰 없이 `POST /api/auth/logout` 호출 시 200이 반환됐다.

**원인 분석:**

```
기존 SecurityConfig
  .requestMatchers("/api/auth/**").permitAll()
                     ↑
              logout도 여기 포함됨
```

`/api/auth/**` 와일드카드로 전체 permitAll을 걸었기 때문에 logout도 인증 없이 통과했다.
토큰이 없으면 `authentication.getName()`이 `"anonymousUser"`를 반환하고, `authService.logout("anonymousUser")`가 실행되지만 DB에 없으니 그냥 200으로 끝났다.

**수정 1 — SecurityConfig 경로 명시 분리:**

`/api/auth/**` 와일드카드를 제거하고 각 경로를 명시적으로 분리했다.

```java
// 수정 전
.requestMatchers("/api/auth/**").permitAll()

// 수정 후
.requestMatchers(
    "/api/auth/signup",
    "/api/auth/login",
    "/api/auth/refresh"
).permitAll()
.requestMatchers("/api/auth/logout").authenticated()
```

`signup`, `login`, `refresh`는 토큰 없이 호출하는 API이므로 `permitAll` 유지.
`logout`은 유효한 Access Token이 있어야 하므로 `authenticated()`로 분리.

> `refresh`를 `permitAll`로 두는 이유 — refresh를 호출하는 상황 자체가 Access Token이 만료된 상태다. `authenticated()`로 막으면 Access Token 없이 호출이 불가능해져서 재발급 자체가 안 된다. 대신 서비스 레이어에서 Refresh Token을 DB 조회 + 만료 시간 + 서명 검증 3단계로 직접 검증한다.

**수정 2 — AuthController logout 메서드 정리:**

`SecurityContextHolder.getContext().getAuthentication()` 직접 꺼내는 방식을 제거하고, 메서드 파라미터로 `Authentication`을 주입받도록 변경했다.

```java
// 수정 전
public ResponseEntity<Void> logout() {
    Authentication authentication = SecurityContextHolder.getContext().getAuthentication();
    String email = authentication.getName();
    authService.logout(email);
    return ResponseEntity.ok().build();
}

// 수정 후
public ResponseEntity<Void> logout(Authentication authentication) {
    authService.logout(authentication.getName());
    return ResponseEntity.ok().build();
}
```

SecurityConfig에서 이미 `authenticated()`로 막으므로 여기까지 오면 인증된 상태가 보장된다. 파라미터 주입 방식이 더 명시적이고 테스트하기도 쉽다.

### JwtAuthenticationEntryPoint 추가 — 401 응답 포맷 통일

**문제:**
logout 수정 후 토큰 없이 호출하면 401이 반환되는데, 이건 Spring Security가 Filter 단계에서 직접 응답을 작성하기 때문에 `GlobalExceptionHandler`를 거치지 않는다.

```
요청
  → FilterChain (Spring Security 영역)
      → AuthorizationFilter — 인증 실패 감지
          → AuthenticationEntryPoint 호출 ← Security가 직접 응답 작성
  → DispatcherServlet (Spring MVC 영역)
      → Controller
      → GlobalExceptionHandler  ← 여기는 도달 안 함
```

결과적으로 다른 API 에러 응답 포맷과 달라진다.

```json
// GlobalExceptionHandler 응답 포맷
{ "code": "INVALID_TOKEN", "message": "유효하지 않은 토큰입니다." }

// Spring Security 기본 응답 포맷
{ "timestamp": "...", "status": 401, "error": "Unauthorized", "path": "..." }
```

**해결 — JwtAuthenticationEntryPoint 구현:**

`AuthenticationEntryPoint`를 구현해서 SecurityConfig에 등록했다. 인증 실패 시 `GlobalExceptionHandler`와 동일한 `ErrorResponse` 포맷으로 응답한다.

```java
@Override
public void commence(HttpServletRequest request,
                     HttpServletResponse response,
                     AuthenticationException authException) throws IOException {

    response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
    response.setContentType(MediaType.APPLICATION_JSON_VALUE);
    response.setCharacterEncoding("UTF-8");

    ErrorResponse body = ErrorResponse.of(ErrorCode.INVALID_TOKEN);
    response.getWriter().write(objectMapper.writeValueAsString(body));
}
```

`SecurityConfig`에 등록:

```java
.exceptionHandling(ex -> ex
    .authenticationEntryPoint(jwtAuthenticationEntryPoint))
```

**수정 후 동작:**

```
토큰 없이 POST /api/auth/logout
  → JwtAuthenticationFilter — SecurityContext 비어있음
  → SecurityConfig .authenticated() → 인증 실패
  → JwtAuthenticationEntryPoint.commence() 호출
  → 응답: 401
    { "code": "INVALID_TOKEN", "message": "유효하지 않은 토큰입니다." }
```

## 깨달은 점 / 막혔던 것

### GlobalExceptionHandler가 Security 예외를 못 잡는 이유

`@RestControllerAdvice`는 DispatcherServlet 내부(Controller 레이어)의 예외만 잡는다. Spring Security Filter는 DispatcherServlet 앞단에서 동작하기 때문에, 인증 실패는 예외를 던지는 게 아니라 `response`에 직접 응답을 작성하고 필터 체인을 끊어버린다. 그래서 Security 레이어의 401/403을 커스터마이징하려면 `AuthenticationEntryPoint`(401)와 `AccessDeniedHandler`(403)를 별도로 구현해야 한다.

### /api/auth/** 와일드카드의 위험성

편의상 와일드카드로 묶으면 나중에 추가되는 엔드포인트도 의도치 않게 허용될 수 있다. 인증이 필요한 엔드포인트는 명시적으로 분리하는 게 안전하다.

## 한투 API 기간별시세 조회 + DB 저장

### 구조

```
stocks        — 종목 마스터 (code, name)
stock_prices  — 일봉 데이터 (stock_id, trade_date, open/high/low/close/volume)
               유니크 제약: (stock_id, trade_date) — 같은 종목 같은 날짜 중복 저장 방지
```

### 흐름

```
POST /api/stocks/005930/fetch?startDate=20240101&endDate=20241231
  → 한투 FHKST03010100 (국내주식 기간별시세) API 호출
  → stocks에 종목 없으면 자동 등록
  → stock_prices에 일봉 저장 (이미 있는 날짜 스킵)
  → 저장된 건수 반환
```

### 트러블슈팅 — FID_ORG_ADJ_PRC 오타

**에러 원문:**
```
ERROR INPUT FIELD NOT FOUND [FID_ORG_ADJ_PRC]
```

**원인:** 파라미터 이름을 `FID_ORG_ADJ_PV`로 잘못 입력했다. 한투 API 스펙상 정확한 이름은 `FID_ORG_ADJ_PRC`.

**해결:** 파라미터명 수정 후 정상 동작. 삼성전자 2024년 일봉 100건 저장 확인.

## 공통 응답 포맷 (ApiResult) 적용

### 문제

성공 응답 포맷이 API마다 달랐다.

```json
// login
{ "accessToken": "...", "refreshToken": "..." }

// fetch
{ "stockCode": "005930", "savedCount": 100 }
```

### 해결 — ApiResult 래퍼 클래스

```java
// 성공 응답: { "success": true, "data": { ... } }
// 데이터 없는 성공: { "success": true }  (data 필드 제외 — @JsonInclude(NON_NULL))
public record ApiResult<T>(boolean success, T data) {
    public static <T> ApiResult<T> ok(T data) { return new ApiResult<>(true, data); }
    public static ApiResult<Void> ok() { return new ApiResult<>(true, null); }
}
```

`ApiResponse`로 먼저 만들었다가 Swagger 어노테이션 `@ApiResponse`와 이름 충돌로 컴파일 에러 발생 → `ApiResult`로 변경.

**적용 후:**

```json
{ "success": true, "data": { "accessToken": "...", "refreshToken": "..." } }
```
