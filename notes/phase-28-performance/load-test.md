# 부하 테스트 종류와 도구

## 부하 테스트란

실제 트래픽과 유사한 환경을 인위적으로 만들어서 시스템의 성능과 한계를 측정하는 테스트.

단순히 "느리다/빠르다"를 확인하는 게 아니라:
- 어느 지점에서 느려지는가
- 어느 지점에서 무너지는가
- 오래 돌리면 어떤 문제가 생기는가

를 수치로 측정하는 것이 목적이다.

---

## 테스트 종류

### 1. Load Test (부하 테스트)
**"정상 트래픽에서 시스템이 버티는가?"**

예상되는 평균 트래픽을 일정 시간 동안 지속적으로 가한다.

```
VU: ──────────────────── (일정)
     시간 →
```

- 목적: 정상 부하 하에서 응답시간, 처리량(TPS), 에러율 측정
- 예: 동시 사용자 100명이 30분 동안 API 호출
- **pong-to-rich 활용: DB vs Redis 성능 비교, 인덱스 유무 비교**

---

### 2. Stress Test (스트레스 테스트)
**"시스템이 언제 무너지는가?"**

부하를 점진적으로 올려서 한계점(breaking point)을 찾는다.

```
VU:     /
       /
      /
     /
    /
     시간 →
```

- 목적: 최대 처리 가능 트래픽 파악, 장애 발생 시점 확인
- 예: 10명 → 50명 → 100명 → 500명 순으로 증가
- **pong-to-rich 활용: HikariCP 커넥션풀 한계 확인**

---

### 3. Spike Test (스파이크 테스트)
**"갑자기 트래픽이 폭발하면?"**

짧은 시간에 급격한 트래픽 증가를 시뮬레이션한다.

```
VU:      |
         |
    ─────┘ └─────
     시간 →
```

- 목적: 이벤트/쿠폰 오픈 같은 순간 폭발 트래픽 대응 확인
- 예: 평소 10명 → 갑자기 1000명 → 다시 10명
- 오토스케일링이 제때 동작하는지 확인하는 데도 쓰인다

---

### 4. Soak Test / Endurance Test (지속 테스트)
**"오래 돌리면 메모리 누수나 커넥션 고갈이 생기는가?"**

낮거나 보통 수준의 부하를 수 시간~수일 동안 지속한다.

```
VU: ─────────────────────────── (낮지만 길게)
     시간 →
```

- 목적: 메모리 누수, DB 커넥션 고갈, 로그 파일 증가 등 장기 운영 문제 발견
- 예: 동시 사용자 50명으로 8시간 연속 실행
- JVM 힙이 서서히 올라가는 패턴이 보이면 메모리 누수 의심

---

### 5. Breakpoint Test
**"절대 한계가 어디인가?"**

Stress Test와 유사하지만 시스템이 완전히 실패(500 에러, 타임아웃)할 때까지 밀어붙인다.

- 목적: 인프라 용량 계획, 스케일 아웃 기준 수립
- 결과를 기준으로 "이 정도 트래픽이면 서버 N대가 필요하다"를 계산

---

## 부하 테스트 도구 비교

| 도구 | 시나리오 언어 | 특징 | 적합한 경우 |
|------|-------------|------|------------|
| **k6** | JavaScript | 가볍고 Grafana 네이티브 연동 | 모던 백엔드 성능 테스트 |
| **JMeter** | GUI / XML | 기업 표준, 무겁고 복잡 | 레거시 환경, 비개발자 포함 팀 |
| **Gatling** | Scala | 리포트 예쁨, JVM 위에서 동작 | Java/Kotlin 프로젝트 |
| **Locust** | Python | 분산 테스트에 강함, 웹 UI 제공 | 대규모 분산 환경 |
| **Artillery** | YAML | 간단하고 빠르게 작성 가능 | 간단한 API 테스트 |

### k6를 선택한 이유

- JavaScript로 시나리오 작성 — 별도 언어 학습 불필요
- CLI 실행 → 결과를 Grafana로 실시간 시각화 가능 (이미 Grafana 세팅 완료)
- 가볍고 컨테이너 친화적
- Load / Stress / Spike 모든 패턴을 하나의 도구로 표현 가능

## 주요 측정 지표

| 지표 | 설명 | 기준 |
|------|------|------|
| **응답시간 P95** | 상위 5% 느린 요청의 응답시간 | 평균보다 P95/P99가 더 중요 |
| **TPS (Throughput)** | 초당 처리 요청 수 | 높을수록 좋음 |
| **에러율** | 전체 요청 중 에러 비율 | 0%에 가까울수록 좋음 |
| **응답시간 평균** | 모든 요청의 평균 응답시간 | 단독으로 보면 오해 가능 |

**P95를 보는 이유:**
평균 응답시간이 100ms여도 5%의 사용자가 2000ms를 경험할 수 있다.
평균은 느린 요청이 희석되어 실제 사용자 경험을 숨긴다.
P95/P99는 "대부분의 사용자가 경험하는 최악의 응답시간"을 나타낸다.

## k6 결과 출력 읽는 법

k6 실행 후 터미널에 출력되는 각 항목의 의미.

### HTTP 섹션

```
http_req_duration..............: avg=1.71s min=220.85ms med=1.7s max=3.43s p(90)=1.99s p(95)=2.96s
```

| 항목 | 설명 |
|------|------|
| `avg` | 전체 요청의 평균 응답시간 |
| `min` | 가장 빠른 요청의 응답시간 |
| `med` | 중앙값 (전체의 정확히 50%가 이 값 이하) |
| `max` | 가장 느린 요청의 응답시간 |
| `p(90)` | 하위 90% 요청이 이 시간 이내에 완료됨 (상위 10%는 이보다 느림) |
| `p(95)` | 하위 95% 요청이 이 시간 이내에 완료됨 (상위 5%는 이보다 느림) |

```
http_req_failed................: 0.00%  0 out of 849
```
- 전체 요청 중 실패(4xx/5xx) 비율과 건수

```
http_reqs......................: 849    26.95/s
```
- 총 요청 수 / 초당 처리 요청 수 (TPS, Throughput)

---

### EXECUTION 섹션

```
iteration_duration.............: avg=1.81s min=321.99ms med=1.8s max=3.53s p(90)=2.09s p(95)=3.06s
iterations.....................: 849    26.95/s
```

- `iteration`: 스크립트의 `default function()` 한 번 실행 = 요청 1번 + sleep
- `iteration_duration`이 `http_req_duration`보다 약간 긴 이유: `sleep(0.1)` (100ms) 포함

```
vus............................: 21     min=21       max=50
vus_max........................: 50     min=50       max=50
```

- `vus`: 테스트 종료 시점의 활성 VU 수 (30s 끝나면서 일부 완료된 상태)
- `vus_max`: 설정한 최대 VU 수 (options에서 지정한 50)

---

### NETWORK 섹션

```
data_received..................: 653 kB 21 kB/s
data_sent......................: 167 kB 5.3 kB/s
```

- 테스트 중 주고받은 총 데이터량 / 초당 데이터량

---

### threshold 결과

```
ERRO thresholds on metrics 'http_req_duration' have been crossed
```

`options`에서 설정한 기준을 넘으면 FAIL로 표시된다.

```js
thresholds: {
  http_req_duration: ['p(95)<500'],  // P95가 500ms 이하여야 함
  error_rate: ['rate<0.01'],         // 에러율 1% 이하여야 함
}
```

threshold FAIL은 "서버 장애"가 아니라 "설정한 성능 목표 미달"을 의미한다.
목표치 자체를 현실에 맞게 조정하거나, 성능 개선 후 다시 측정한다.

## DB vs Redis 로그인 부하 테스트 결과

### 테스트 환경
- 엔드포인트: `POST /api/auth/login`
- VUs: 50 / 지속시간: 30s
- 더미 유저: test001~test100@pongtest.com (비밀번호: Test1234!)
- token.store: `db`

### 1회차 (DB 모드)

```
http_req_duration: avg=1.77s  min=211.82ms  med=1.74s  max=4.77s  p(90)=2.11s  p(95)=2.99s
http_req_failed:   0.85%  (7 out of 820)
http_reqs:         820    26.06/s
```

- 첫 실행이라 RefreshToken이 없어서 INSERT 발생
- 에러 7건 발생 (로그인 경합 or 타임아웃)

### 2회차 (DB 모드)

```
http_req_duration: avg=1.71s  min=220.85ms  med=1.70s  max=3.43s  p(90)=1.99s  p(95)=2.96s
http_req_failed:   0.00%  (0 out of 849)
http_reqs:         849    26.95/s
```

- 이미 토큰이 존재해서 UPDATE 발생 (INSERT → UPDATE)
- 에러 0건, 1회차와 성능 차이 거의 없음

### 분석

| | 1회차 | 2회차 |
|---|---|---|
| P95 | 2.99s | 2.96s |
| TPS | 26.1/s | 26.9/s |
| 에러율 | 0.85% | 0% |

**결과가 느린 이유: BCrypt**

로그인 플로우에서 BCrypt 검증이 병목이다.

```
요청 수신
  → DB 유저 조회
  → BCrypt.verify(입력값, 저장된해시)  ← 요청당 ~100~200ms 소요
  → JWT 생성
  → RefreshToken 저장 (DB or Redis)   ← 비교하고 싶은 부분
  → 응답
```

BCrypt는 해킹 방어를 위해 의도적으로 느리게 설계된 알고리즘이다.
50 VUs가 동시에 BCrypt 연산을 수행하면 스레드 풀이 포화되어 대기 시간이 급증한다.

**DB vs Redis 순수 비교가 필요하다면:**
`/api/auth/refresh` 엔드포인트로 테스트해야 한다.
RefreshToken 조회만 하므로 BCrypt 없이 DB 쿼리 vs Redis GET 차이만 측정된다.

## pong-to-rich에서 할 테스트 계획

| 목적 | 테스트 종류 | 시나리오 |
|------|------------|---------|
| DB vs Redis Refresh Token 성능 비교 | Load Test | 동일 VU로 로그인 반복, P95 응답시간 비교 |
| 인덱스 유무 성능 비교 | Load Test | stock_prices 조회 반복, EXPLAIN + 응답시간 비교 |
| HikariCP 커넥션풀 한계 확인 | Stress Test | VU 점진적 증가, 커넥션 고갈 시점 Grafana로 확인 |
