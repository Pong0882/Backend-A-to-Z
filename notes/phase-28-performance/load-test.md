# 부하 테스트 종류와 도구

## 부하 테스트란

실제 트래픽과 유사한 환경을 인위적으로 만들어서 시스템의 성능과 한계를 측정하는 테스트.

단순히 "느리다/빠르다"를 확인하는 게 아니라:
- 어느 지점에서 느려지는가
- 어느 지점에서 무너지는가
- 오래 돌리면 어떤 문제가 생기는가

를 수치로 측정하는 것이 목적이다.

---

## 테스트 종류

### 1. Load Test (부하 테스트)
**"정상 트래픽에서 시스템이 버티는가?"**

예상되는 평균 트래픽을 일정 시간 동안 지속적으로 가한다.

```
VU: ──────────────────── (일정)
     시간 →
```

- 목적: 정상 부하 하에서 응답시간, 처리량(TPS), 에러율 측정
- 예: 동시 사용자 100명이 30분 동안 API 호출
- **pong-to-rich 활용: DB vs Redis 성능 비교, 인덱스 유무 비교**

---

### 2. Stress Test (스트레스 테스트)
**"시스템이 언제 무너지는가?"**

부하를 점진적으로 올려서 한계점(breaking point)을 찾는다.

```
VU:     /
       /
      /
     /
    /
     시간 →
```

- 목적: 최대 처리 가능 트래픽 파악, 장애 발생 시점 확인
- 예: 10명 → 50명 → 100명 → 500명 순으로 증가
- **pong-to-rich 활용: HikariCP 커넥션풀 한계 확인**

---

### 3. Spike Test (스파이크 테스트)
**"갑자기 트래픽이 폭발하면?"**

짧은 시간에 급격한 트래픽 증가를 시뮬레이션한다.

```
VU:      |
         |
    ─────┘ └─────
     시간 →
```

- 목적: 이벤트/쿠폰 오픈 같은 순간 폭발 트래픽 대응 확인
- 예: 평소 10명 → 갑자기 1000명 → 다시 10명
- 오토스케일링이 제때 동작하는지 확인하는 데도 쓰인다

---

### 4. Soak Test / Endurance Test (지속 테스트)
**"오래 돌리면 메모리 누수나 커넥션 고갈이 생기는가?"**

낮거나 보통 수준의 부하를 수 시간~수일 동안 지속한다.

```
VU: ─────────────────────────── (낮지만 길게)
     시간 →
```

- 목적: 메모리 누수, DB 커넥션 고갈, 로그 파일 증가 등 장기 운영 문제 발견
- 예: 동시 사용자 50명으로 8시간 연속 실행
- JVM 힙이 서서히 올라가는 패턴이 보이면 메모리 누수 의심

---

### 5. Breakpoint Test
**"절대 한계가 어디인가?"**

Stress Test와 유사하지만 시스템이 완전히 실패(500 에러, 타임아웃)할 때까지 밀어붙인다.

- 목적: 인프라 용량 계획, 스케일 아웃 기준 수립
- 결과를 기준으로 "이 정도 트래픽이면 서버 N대가 필요하다"를 계산

## 부하 테스트 도구 비교

| 도구 | 시나리오 언어 | 특징 | 적합한 경우 |
|------|-------------|------|------------|
| **k6** | JavaScript | 가볍고 Grafana 네이티브 연동 | 모던 백엔드 성능 테스트 |
| **JMeter** | GUI / XML | 기업 표준, 무겁고 복잡 | 레거시 환경, 비개발자 포함 팀 |
| **Gatling** | Scala | 리포트 예쁨, JVM 위에서 동작 | Java/Kotlin 프로젝트 |
| **Locust** | Python | 분산 테스트에 강함, 웹 UI 제공 | 대규모 분산 환경 |
| **Artillery** | YAML | 간단하고 빠르게 작성 가능 | 간단한 API 테스트 |

### k6를 선택한 이유

- JavaScript로 시나리오 작성 — 별도 언어 학습 불필요
- CLI 실행 → 결과를 Grafana로 실시간 시각화 가능 (이미 Grafana 세팅 완료)
- 가볍고 컨테이너 친화적
- Load / Stress / Spike 모든 패턴을 하나의 도구로 표현 가능

## 주요 측정 지표

| 지표 | 설명 | 기준 |
|------|------|------|
| **응답시간 P95** | 상위 5% 느린 요청의 응답시간 | 평균보다 P95/P99가 더 중요 |
| **TPS (Throughput)** | 초당 처리 요청 수 | 높을수록 좋음 |
| **에러율** | 전체 요청 중 에러 비율 | 0%에 가까울수록 좋음 |
| **응답시간 평균** | 모든 요청의 평균 응답시간 | 단독으로 보면 오해 가능 |

**P95를 보는 이유:**
평균 응답시간이 100ms여도 5%의 사용자가 2000ms를 경험할 수 있다.
평균은 느린 요청이 희석되어 실제 사용자 경험을 숨긴다.
P95/P99는 "대부분의 사용자가 경험하는 최악의 응답시간"을 나타낸다.

## pong-to-rich에서 할 테스트 계획

| 목적 | 테스트 종류 | 시나리오 |
|------|------------|---------|
| DB vs Redis Refresh Token 성능 비교 | Load Test | 동일 VU로 로그인 반복, P95 응답시간 비교 |
| 인덱스 유무 성능 비교 | Load Test | stock_prices 조회 반복, EXPLAIN + 응답시간 비교 |
| HikariCP 커넥션풀 한계 확인 | Stress Test | VU 점진적 증가, 커넥션 고갈 시점 Grafana로 확인 |
